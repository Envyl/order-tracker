# Feature Specification: Order Tracker

**Feature Branch**: `001-order-tracker`

**Created**: 2026-07-18

**Status**: Draft

**Input**: User description: "display name = Order Tracker; уточни способы авторизации для каждого провайдера; уже есть учётки — в приложении нужна возможность авторизоваться к ним имеющимися учётками, не нужна (отдельная) авторизация; интерфейс простой и понятный; должно быть понятно, какой провайдер отображает статус и как выглядит товар в заказе"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Связать существующие аккаунты провайдеров (Priority: P1)

Пользователь открывает Order Tracker на iPhone и по очереди подключает уже существующие личные аккаунты Wildberries, AliExpress и СДЭК. В приложении нет отдельной регистрации и входа в «аккаунт Order Tracker» — только привязка учёток провайдеров через их обычные учётные данные.

**Why this priority**: Без подключённых аккаунтов агрегировать заказы нельзя; это фундамент продукта.

**Independent Test**: Можно полностью проверить, подключив хотя бы один провайдер с валидными данными и убедившись, что связь сохраняется между запусками приложения, без создания аккаунта Order Tracker.

**Acceptance Scenarios**:

1. **Given** приложение только что установлено и ни один провайдер не подключён, **When** пользователь открывает экран подключений, **Then** он видит три провайдера (Wildberries, AliExpress, СДЭК) со статусом «не подключено» и может начать подключение любого из них.
2. **Given** пользователь подключает Wildberries, **When** он вводит учётные данные существующего аккаунта Wildberries (телефон и код/пароль в том виде, который требует Wildberries для входа в личный кабинет), **Then** приложение сохраняет успешную привязку и показывает Wildberries как «подключено».
3. **Given** пользователь подключает AliExpress, **When** он вводит учётные данные существующего аккаунта AliExpress (email или телефон и пароль, плюс подтверждение, если его запросит AliExpress), **Then** приложение сохраняет успешную привязку и показывает AliExpress как «подключено».
4. **Given** пользователь подключает СДЭК, **When** он вводит учётные данные существующего личного кабинета СДЭК (телефон/логин и код или пароль в том виде, который требует СДЭК), **Then** приложение сохраняет успешную привязку и показывает СДЭК как «подключено».
5. **Given** пользователь неверно ввёл данные любого провайдера, **When** привязка не удалась, **Then** приложение показывает понятное сообщение об ошибке, не помечает провайдера как подключённого и позволяет повторить попытку.
6. **Given** хотя бы один провайдер подключён, **When** пользователь полностью закрывает и снова открывает приложение, **Then** привязки остаются без повторного ввода данных.

### User Story 2 - Единый список статусов заказов (Priority: P1)

Пользователь открывает главный экран и видит один простой список заказов со всех подключённых провайдеров. Для каждой позиции сразу понятно: откуда статус (провайдер), что за товар (название и внешний вид), и какой сейчас статус доставки/заказа.

**Why this priority**: Это основная ценность приложения — не открывать три разных сервиса, чтобы понять, где заказы.

**Independent Test**: При подключённых тестовых аккаунтах (или подготовленных данных) пользователь одним взглядом на список понимает провайдера, товар и статус без перехода в сторонние приложения.

**Acceptance Scenarios**:

1. **Given** подключены один или несколько провайдеров с активными заказами, **When** пользователь открывает главный экран (или обновляет список), **Then** он видит единый список заказов от всех успешно ответивших провайдеров.
2. **Given** в списке есть заказ, **When** пользователь смотрит карточку заказа, **Then** на ней явно указан провайдер (Wildberries / AliExpress / СДЭК), показаны название товара и его изображение (или явный placeholder, если картинки нет), а также понятный статус.
3. **Given** пользователь хочет актуализировать данные, **When** он запускает обновление, **Then** статусы и сведения о товарах обновляются, а время последнего успешного обновления видно на экране.
4. **Given** у пользователя нет активных заказов ни у одного подключённого провайдера, **When** он открывает список, **Then** он видит понятное пустое состояние с подсказкой, а не «сломанный» экран.

---

### User Story 3 - Простой просмотр деталей заказа (Priority: P2)

Пользователь открывает заказ из списка и видит детали в том же простом стиле: провайдер, товар (название + изображение), текущий статус, идентификатор заказа/отправления и дата последнего обновления статуса.

**Why this priority**: Уточняет информацию из списка, но не блокирует базовую ценность единого списка.

**Independent Test**: Открытие любого заказа из списка показывает детали без необходимости заходить в приложение провайдера.

**Acceptance Scenarios**:

1. **Given** пользователь на главном списке, **When** он выбирает заказ, **Then** открывается экран деталей с провайдером, изображением и названием товара, статусом, идентификатором и временем обновления.
2. **Given** статус у провайдера неизвестен или устарел, **When** пользователь открывает детали, **Then** приложение явно показывает, что данные неполные или устаревшие, а не выдумывает статус.

---

### User Story 4 - Устойчивость при сбое одного провайдера (Priority: P2)

Если один провайдер недоступен или сессия истекла, остальные продолжают отображаться. Пользователь понимает, какой источник не обновился, и может заново подключить учётку.

**Why this priority**: Реальная работа с тремя внешними сервисами неизбежно включает частичные сбои.

**Independent Test**: Отключить или «сломать» один провайдер и убедиться, что заказы других всё ещё видны, а проблемный источник помечен понятной ошибкой.

**Acceptance Scenarios**:

1. **Given** подключены все три провайдера, **When** один из них не отвечает при обновлении, **Then** заказы остальных остаются в списке, а для сбойного провайдера показывается понятное предупреждение.
2. **Given** сессия провайдера истекла, **When** пользователь пытается обновить данные этого провайдера, **Then** приложение просит заново ввести учётные данные этого провайдера и не требует «входа в Order Tracker».

---

### Edge Cases

- Что происходит, если пользователь подключает только один или два провайдера из трёх?
- Что происходит, если у заказа несколько товаров — как показать «как выглядит товар» (первый товар / сводка)?
- Что происходит, если провайдер не отдаёт изображение товара?
- Что происходит, если один провайдер недоступен, а другие отвечают нормально?
- Что происходит, если учётные данные верны, но у аккаунта нет заказов?
- Что происходит при повторной привязке уже подключённого провайдера (замена сессии)?
- Что происходит, если локальная установка на iPhone ещё не выполнена / приложение переустановлено (привязки сбрасываются)?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Приложение MUST отображаться пользователю под именем **Order Tracker**.
- **FR-002**: Система MUST NOT требовать отдельной регистрации или входа в аккаунт Order Tracker.
- **FR-003**: Пользователь MUST иметь возможность подключить существующий аккаунт Wildberries, введя учётные данные, которые он уже использует для входа в Wildberries (телефон и код подтверждения или пароль — в соответствии с тем, что требует Wildberries для личного кабинета).
- **FR-004**: Пользователь MUST иметь возможность подключить существующий аккаунт AliExpress, введя учётные данные, которые он уже использует для входа в AliExpress (email или телефон и пароль, плюс дополнительное подтверждение, если его запрашивает AliExpress).
- **FR-005**: Пользователь MUST иметь возможность подключить существующий личный кабинет СДЭК, введя учётные данные, которые он уже использует для входа в СДЭК (телефон/логин и код или пароль — в соответствии с тем, что требует СДЭК).
- **FR-006**: Система MUST NOT предлагать создание новых аккаунтов у провайдеров внутри Order Tracker; только привязка уже существующих учёток.
- **FR-007**: Система MUST сохранять успешные привязки провайдеров между запусками приложения на том же устройстве.
- **FR-008**: Система MUST показывать единый список заказов по всем подключённым провайдерам.
- **FR-009**: Для каждой позиции списка система MUST явно показывать источник статуса: Wildberries, AliExpress или СДЭК.
- **FR-010**: Для каждой позиции списка система MUST показывать, как выглядит товар в заказе: название и изображение товара; если изображения нет — явный placeholder «нет фото».
- **FR-011**: Для каждой позиции списка система MUST показывать текущий понятный статус заказа/доставки.
- **FR-012**: Пользователь MUST иметь возможность вручную обновить статусы.
- **FR-013**: Система MUST сохранять и показывать последние известные статусы, если обновление частично не удалось.
- **FR-014**: Сбой или истечение сессии одного провайдера MUST NOT скрывать заказы остальных провайдеров.
- **FR-015**: Пользователь MUST иметь возможность отключить провайдера и заново привязать его учётные данные.
- **FR-016**: Интерфейс MUST быть простым: главный экран — список заказов; отдельный понятный поток — подключение провайдеров; без лишних разделов, лент и маркетинговых блоков.
- **FR-017**: Если в заказе несколько товаров, система MUST показать изображение и название основного (первого) товара и указать, что в заказе есть ещё позиции (например, «и ещё N»).
- **FR-018**: Пользователь MUST иметь возможность открыть детали заказа с провайдером, товаром (название + изображение), статусом, идентификатором и временем последнего обновления.

### Key Entities

- **ProviderConnection**: Привязка существующего аккаунта провайдера (Wildberries / AliExpress / СДЭК); статус подключения; без сущности «пользователь Order Tracker».
- **Order**: Заказ или отправление в едином списке; принадлежит одному провайдеру; имеет нормализованный статус и время обновления.
- **OrderItem**: Товар в заказе; название и изображение (внешний вид); один из товаров может быть основным для превью в списке.
- **StatusSnapshot**: Зафиксированное состояние статуса на момент последнего успешного получения данных от провайдера.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: Пользователь подключает существующий аккаунт каждого из трёх провайдеров менее чем за 3 минуты на провайдера при известных учётных данных (без создания аккаунта Order Tracker).
- **SC-002**: После обновления пользователь видит в одном списке заказы со всех успешно ответивших подключённых провайдеров.
- **SC-003**: В слепом просмотре 10 случайных карточек заказов пользователь в ≥90% случаев с первого взгляда правильно называет провайдера и узнаёт товар по изображению/названию.
- **SC-004**: При недоступности одного провайдера пользователь всё ещё видит заказы остальных без перезапуска приложения.
- **SC-005**: Первичная проверка статуса («где мои заказы?») выполняется без открытия приложений Wildberries, AliExpress или СДЭК в ≥95% успешных сценариев при валидных привязках.
- **SC-006**: Новый пользователь по quickstart устанавливает Order Tracker на физический iPhone с домашнего компьютера и доходит до экрана списка заказов менее чем за 30 минут.

## Assumptions

- Отображаемое имя продукта — **Order Tracker** (имя репозитория/проекта Jules на это не влияет).
- Отдельный аккаунт Order Tracker не нужен и не планируется в этой фиче.
- У пользователя уже есть личные аккаунты Wildberries, AliExpress и/или СДЭК; приложение только привязывает их.
- Способы входа по провайдерам (пользовательский уровень):
  - **Wildberries** — телефон + код/пароль личного кабинета Wildberries;
  - **AliExpress** — email или телефон + пароль (+ подтверждение по запросу AliExpress);
  - **СДЭК** — телефон/логин + код или пароль личного кабинета СДЭК.
- Подключение трекинг-номером без аккаунта СДЭК в эту фичу не входит (фокус на существующих кабинетах).
- В списке показываются активные и недавно завершённые заказы (горизонт «недавние» — порядка последних 30 дней), а не полный архив за все годы.
- Целевое устройство — iPhone; первичная установка — локально с домашнего компьютера (не App Store-first).
- Один пользователь / одно устройство в baseline; несколько профилей Order Tracker не нужны.
- Язык интерфейса по умолчанию — русский.
- Если провайдер отдаёт несколько изображений товара, достаточно одного основного превью.
